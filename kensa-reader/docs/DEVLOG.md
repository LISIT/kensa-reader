# 開発の軌跡 — 意思決定ログ・制約マップ・設計原則

このドキュメントは、kensa-reader（やさしい検査結果リーダー / Medical AI Navigator）の
開発で実際にぶつかった**紆余曲折・挫折・方針転換**を一次資料として記録するものです。
目的は、**同じ轍を踏まないこと**、そして「**1人＋AIエージェント**」での開発過程の
ケーススタディとして残すことです（無償・オープンソース）。

> 要旨: 「自前OCRで完結する医療アプリ」を目指して何度も壁にぶつかり、最終的に
> 「**フロンティアAI（ChatGPT/Claude/Gemini）への橋渡しに徹するナビゲーター**」へ
> 設計転換した。その過程で、モバイルWebの再現可能な制約群を実機で洗い出した。

| 3 | 初期実装 | 血液検査の OCR(Tesseract)＋内蔵基準値ナレッジで判定 | 数値表は決定論的判定が確実、と想定 |
| 4 | 挫折① | 実画像でほぼ読めず（横向き・虹色背景・小さい文字） | Tesseractは雑なスマホ写真・日本語に弱い |

| 7 | 検証 | 手元ハーネスで実画像比較。**PaddleOCR >> Tesseract** を実証 | 英語は小数・基準値まで正確、日本語漢字も読む |
| 8 | 挫折② | PaddleOCR(onnxruntime-web)を**iOS Safariがメモリ不足でクラッシュ** | スマホブラウザで重いOCRは不安定 |
| 9 | 現実解 | OCRは補助に降格。**手入力補助UI**（読めた分は自動、読めない分だけ手入力） | 完全自動を諦め、ユーザー確認を前提に |
| 10 | **大転換** | **Medical AI Navigator** へ。自前OCRでなく**既存AIへの橋渡し**に徹する | ChatGPT等は画像も検査票も十分読める。問題はAI性能でなく「使い方が分からない」UX |
| 11 | 矛盾の露呈 | 「自動・AI不可視」と「無料・キーなし・課金なし」は**同時成立不可** | 自動回答取得はキー/バックエンド課金が必須。ユーザーは後者を優先 |
| 12 | 挫折③ | Web共有後に **iOS Safari がフリーズ**。flushSync/再読込/ポーリング**全て失敗** | `navigator.share` がWebView単位でタッチを固める（再読込でも直らない） |
| 13 | ネイティブ化 | **Capacitor**（@capacitor/share）でフリーズ回避を決定 | ネイティブ共有はWebViewのnavigator.shareと別物 |
| 14 | 制約 | iOSビルドは**macOS必須**。WindowsのみではiPhone不可 | Mac or クラウドCI+Apple Developer($99) |
| 15 | 再評価 | **Webリンクを主役に戻す**（フリーズは許容） | 「リンクを開くだけ」の簡単さが高齢者には最優先。フリーズは“答えを得た後”で実害小 |
| 16 | 端末依存の判明 | iPhone14Pro=固まる/自動ペースト、**16Pro=固まらない/手動**、Android=手動 | 「iOSなら〜」の出し分けは不可 → 図解は全機種で出す |
| 17 | UX磨き込み | 図解は**送信の“前”**に置く（戻ってからでは無意味）。ボタン下にAIアイコン＋「から選択」 | 案内は行動の前に。選択肢を視覚化 |

## 2. 制約マップ（再現可能な技術的制約）

実機・実装で確認した、**回避不能または条件付きの制約**。

### モバイルWeb / iOS
- **`navigator.share` 後のフリーズ（iOS Safari）**: 共有シートが描画に割り込み、共有後に
  **WebView単位でタッチが不応答**になる。`location.reload()` でも直らない（ページ単位でなく
  WebView単位）。flushSync・遅延・pageshow/visibilitychangeポーリング、すべて無効。
  → 回避はネイティブ共有(@capacitor/share)のみ。または「フリーズ許容＋送信後は静的画面」。
- **クリップボード自動挿入は端末依存**: iPhone14Pro は共有後にChatGPTへ質問が自動で入る
  （iOSのクリップボード自動挿入）。**iPhone16Pro / Android は入らない（手で長押し→ペースト）**。
  同一OS内でも世代差で逆になる → OSやUAでの出し分けは信頼できない。
- **`transient activation`**: `navigator.share`/`window.open` の前に `await` を挟むと
  iOSで操作有効性が切れ、貼り付け不能や不発を招く。**コピー等は共有/オープンの“前に同期発火”**。

### オンデバイス推論
- **onnxruntime-web(PaddleOCR) は iOS Safari でメモリ不足クラッシュ**: 10MB級JS＋wasm＋
  モデル15MBの読込でタブが落ちる。スマホブラウザで重い推論は不安定。
- **esm.sh `?bundle` の unenv**: `process.binding is not implemented yet!` で実行時失敗。
  → Vite側で正しくブラウザ向けにバンドルすべき（CDNバンドルに頼らない）。

### 共有先AIアプリ
- **ChatGPT等は共有された画像のみ取り込み、付随テキストを無視**することが多い。
  → 質問文の自動添付は不可。クリップボード経由の手貼りが基本。

### OCR
- **Tesseract** は日本語・雑なスマホ写真（傾き・背景・小文字）に弱い。**PaddleOCR/RapidOCR**は
  日英・小数・基準値まで強いが、ブラウザ実行は重くiOSで不安定。
- 検査票は**多パネル・多列レイアウト**で、bboxの単純なy行まとめが崩れる → 列認識が要る。

### ビルド/配布
- **Windows → iOS は macOS/Xcode 必須**。実機配布は Apple Developer（$99/年, TestFlight）か
  Mac＋無料7日署名。Androidは Windows + Android Studio で可。

### 設計レベルの「両立不可」
- 「**自動・AIを見せない**」×「**無料・キーなし・課金なし**」は同時成立しない
  （自動回答取得はキー/バックエンド課金が必須）。どちらを優先するかを先に決めるべき。

## 3. 設計原則（得られた教訓）

1. **自前AIを作らず、フロンティアAIへの橋渡しに徹する**（"navigator, not engine"）。
   性能はフロンティアAIに任せ、アプリは「最適なプロンプト生成＋受け渡し導線＋安全注意」に集中。
2. **アクセスの簡単さが最優先**（高齢者・低ITリテラシー）。「リンクを開くだけ」に勝る入口はない。
   ネイティブの滑らかさより、配布・更新の容易さ（Webリンク）が効く。
3. **案内は“行動の前”に出す**。共有後・遷移後に出る案内は手遅れで無意味。
4. **完全自動を狙わない**。OCRは補助。**ユーザー確認済みの値だけ**をレポートに使う。
5. **技術用語をUIに出さない**（API/LLM/OCR/Prompt/JSON）。AI名（ChatGPT等）は安心感のため可。
6. **安全第一・非薬事**: 診断しない／断定しない（がん・良性等の禁止表現）／必ず医師へ／
   緊急時は救急。プロンプトにも安全指示を埋め込む。
7. **実機・複数端末で検証**。机上では分からない端末依存（同一OSの世代差）が体験を左右する。
8. **「両立不可」を見極めて優先順位を先に決める**。トレードオフを曖昧にしたまま実装しない。
9. **制約は回避策とセットで記録**（このドキュメント自体がその実践）。

## 4. 同じ轍を踏まないためのチェックリスト

- [ ] スマホ写真の文字認識を「アプリだけで完結」しようとしていないか（→ フロンティアAIに委ねる）
- [ ] iOSで `navigator.share` 後のフリーズを前提に、**送信後は静的な完了/案内画面**にしているか
- [ ] 共有/オープンの前に `await` を挟んでいないか（クリップボードコピーは同期発火）
- [ ] 自動ペーストを前提にしていないか（端末依存。**手貼り導線と図解を“送信前”に**用意）
- [ ] 「無料・キーなし」と「自動・不可視」を同時に約束していないか
- [ ] onnxruntime等の重い推論をモバイルブラウザ既定にしていないか（iOSメモリ）
- [ ] UIに技術用語を出していないか／安全注意（非診断・医師へ）を常時表示しているか
- [ ] 実機を**複数世代**（例: iPhone旧/新, Android）で確認したか

## 5. 「Institution of One」ケースとしての位置づけ

本プロジェクトは、**1人の意思決定者＋AIコーディングエージェント**という最小単位で、
構想 → 数次の大方針転換 → 公開・実機検証 → 制約の体系化 まで短時間で到達した事例である。

- **人間が担った不可欠な判断**: OCR断念の決断、「アクセス簡単さ ＞ フリーズ回避」の優先、
  案内配置（送信前）の発見、そして「これは論文になるか」というメタ判断。
- **エージェントが担った領域**: 実装・調査・反復・制約の検証ハーネス作成・ドキュメント化。
- **一次資料**: 本リポジトリの git 履歴が、そのまま時系列の意思決定ログになっている。

無償・オープンソースのユニーク開発事例として、IORN（Institute of One Research Network /
https://instituteofone.org/ ）等の番外ケーススタディに供しうる。
※ 外部団体の要件・採否は未確認。本節は当プロジェクト側の位置づけの記述にとどめる。
